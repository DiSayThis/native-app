# AGENTS.md

## Цель

Этот файл задает правила для AI-агентов (Codex и аналогов), чтобы изменения в проекте были предсказуемыми, переиспользуемыми и совместимыми с текущим стеком Expo + React Native + FSD.

## Обязательный Workflow Перед Любым Изменением

1. Проверить существующую структуру и найти, где уже решена похожая задача.
2. Проверить уже установленные пакеты в `package.json` и использовать их, если они покрывают задачу.
3. Проверить, можно ли переиспользовать компоненты из `shared` или `widgets` вместо создания новых.
4. Только если переиспользование невозможно, создавать новый компонент/хук/утилиту в правильном слое FSD.
5. Любые UI-изменения должны корректно работать в светлой и темной теме.

## Карта Проекта (Где Что Искать)

- `app/`: роутинг и верхнеуровневые layout-файлы Expo Router, глобальные провайдеры.
- `pages/`: экранная композиция (сборка экрана из `widgets/features/entities/shared`).
- `widgets/`: крупные переиспользуемые блоки экрана (секции/композитные UI-блоки).
- `features/`: пользовательские сценарии и action-flow (форма логина, регистрация, избранное и т.д.).
- `entities/`: доменные сущности (user/partner/auth/city-region), их модели и UI.
- `shared/`: общий UI-kit, хелперы, стили, провайдеры, API-база, типы, ассеты.

Пути-алиасы:

- `@/app/*`, `@/pages/*`, `@/widgets/*`, `@/features/*`, `@/entities/*`, `@/shared/*`

## Правила Переиспользования Компонентов

- Для инпутов и полей форм сначала искать в:
  - `shared/ui/inputs/*`
  - `shared/ui/Input.tsx`
- Для кнопок, модалок и типовых UI-примитивов сначала искать в:
  - `shared/ui/*`
- Для повторяющихся составных секций (карточки, списки, блоки экрана) сначала искать в:
  - `widgets/*`
- Если нужный UI уже есть, расширять его через пропсы/варианты, а не создавать дубликат.

## Когда Выносить Куски В Отдельные Компоненты

Вынос обязателен, если выполняется хотя бы одно условие:

1. Один и тот же JSX/логика повторяется в 2+ местах.
2. Компонент совмещает слишком много ответственности (UI + data-fetch + сложная бизнес-логика).
3. Компонент стал трудно читать/поддерживать (большие условные ветки, длинные секции верстки, смешение уровней абстракции).

Куда выносить:

- Переиспользуемый примитив -> `shared/ui`.
- Переиспользуемый блок экрана из нескольких частей -> `widgets`.
- Логику пользовательского сценария -> `features/<slice>/model|hook|api`.
- Логику доменной сущности -> `entities/<slice>/model|hook|api`.

## Правила Темизации (2 Темы Обязательно)

- В проекте уже есть централизованная тема:
  - `shared/ui/theme/ThemeProvider.tsx`
  - `shared/ui/theme/theme.store.ts`
  - `shared/styles/tokens.ts`
- Все новые компоненты должны использовать `useTheme()` и theme-tokens.
- Не хардкодить цвета/типографику, если можно взять из `theme.colors`, `theme.typography`, `theme.spacing`, `theme.radius`.
- Любой новый UI-элемент должен выглядеть и быть читаемым в `light` и `dark`.

## Правила По Пакетам И Зависимостям

- Перед добавлением новой библиотеки:
  1. проверить `package.json` (возможно пакет уже установлен);
  2. проверить, есть ли решение на текущем стеке (`expo`, `react-native`, текущие зависимости);
  3. проверить совместимость библиотеки с Expo/React Native.
- Для установки/обновления зависимостей использовать `npx expo install`, а не прямой `npm install`/`yarn add`, чтобы Expo подобрал совместимую версию.

## FSD-Границы (Обязательно Соблюдать)

Допустимое направление зависимостей только вниз:

- `app -> pages -> widgets -> features -> entities -> shared`

Запрещено:

- импортировать код вверх по слоям;
- тащить бизнес-логику в `app/pages`;
- складывать бизнес-специфику в `shared`.

## Expo Go + React Native Best Practices (Официальные)

1. Expo Go использовать как быстрый sandbox/прототипирование; для production-grade разработки ориентироваться на development builds.
2. Если библиотека требует native code или не поддерживается Expo Go, переходить на development build.
3. При изменении app config или добавлении native dependency использовать `npx expo prebuild --clean` для регенерации native-проектов.
4. Предпочитать config plugins вместо ручных правок `android/` и `ios/`, чтобы сохранить управляемость CNG-потока.
5. Всегда устанавливать совместимые версии пакетов через `npx expo install`.
6. Производительность проверять не только в dev-режиме, но и в release-сборках.
7. Для больших списков применять практики оптимизации RN (`FlatList`, `getItemLayout` и т.д.).
8. Не оставлять шумные `console.*` в production-бандле.

## Минимальный Чеклист Перед Сдачей Изменений

1. Нет дублирующих компонентов там, где можно было переиспользовать `shared/widgets`.
2. Новая функциональность лежит в корректном FSD-слое.
3. UI корректен в светлой и темной темах.
4. Не добавлены лишние зависимости, если задача решается текущим стеком.
5. Проект проходит линт (`npm run lint`) и форматирование (при необходимости `npm run lint:format-fix`).
